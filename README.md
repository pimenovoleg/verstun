# Verstun

[English](README.en.md)

Verstun - self-host Telegram-бот для авторов каналов. Он берет Markdown-файл,
собирает из него Telegram Rich Message, сохраняет встроенные картинки на вашем
сервере и возвращает готовый пост в личный чат. После проверки пост можно
переслать вручную или опубликовать напрямую в подключенный канал.

Лучший рабочий процесс - писать пост в Google Docs и скачивать его через
`File -> Download -> Markdown (.md)`. Google Docs удобно держит структуру
лонгрида, таблицы, списки и картинки, а Verstun превращает экспорт в Rich
Message. При этом бот не привязан к Google Docs: он принимает обычные `.md`
файлы из других редакторов и источников, если внутри валидный Markdown.

## Как Это Работает

1. Автор отправляет боту `.md` файл в личный чат.
2. Бот парсит Markdown и собирает HTML-источник для Telegram Rich Message.
3. Встроенные `data:` картинки из Markdown декодируются, кладутся в `MEDIA_DIR`
   и становятся публичными HTTPS URL вида `/media/<hash>.<ext>`.
4. Telegram получает Rich Message от бота и сам забирает картинки по этим URL.
5. Бот отправляет автору готовый Rich Message и отдельное сообщение с кнопками
   публикации в доступные каналы.

Поддерживаются заголовки, списки, таблицы, цитаты, код, ссылки, спойлеры,
раскрывающиеся блоки, разделители, сноски, LaTeX-формулы, встроенные
изображения и внешние ссылки на видео, GIF и аудио.

LaTeX пишется обычным Markdown-синтаксисом:

```md
Inline: $E = mc^2$

$$
x = \frac{-b \pm \sqrt{b^2 - 4ac}}{2a}
$$
```

Внешние медиа вставляются тем же Markdown-синтаксисом, что и картинки:

```md
![](https://example.com/photo.jpg)
![](https://example.com/animation.gif)
![](https://example.com/video.mp4)
![](https://example.com/iphone-video.mov)
![](https://example.com/audio.mp3)
![](https://example.com/iphone-audio.m4a)
```

Поддерживаемые внешние форматы определяются только по расширению URL path:
`jpg`, `jpeg`, `png`, `webp`, `heic`, `heif`, `gif`, `mp4`, `mov`, `m4v`,
`webm`, `mp3`, `ogg`, `oga`, `m4a`, `aac`, `wav`, `flac`. Ссылки без
расширения и опасные форматы вроде `svg`, `html`, `pdf`, `exe`, `js`, архивов
или плейлистов не попадают в Rich Message.

Runtime-состояние хранится в JSON-файле в `DATA_DIR`: подключенные каналы,
пользователи, доступы пользователей к каналам, флаг подключения канала, активные
кнопки публикации и кэш сгенерированных постов. Отдельная база данных не нужна.

### Картинки, Видео, GIF И Аудио

Медиа-ссылка должна быть отдельным блоком, то есть на своей строке и с пустыми
строками вокруг:

```md
![](https://example.com/photo.jpg)
![](https://example.com/animation.gif)
![](https://example.com/video.mp4)
![](https://example.com/iphone-video.mov)
![](https://example.com/audio.mp3)
![](https://example.com/iphone-audio.m4a)
```

Подпись можно добавить через Markdown title:

```md
![](https://example.com/video.mp4 "Подпись к видео")
```

Если media-ссылку вставить внутрь обычного текста, таблицы или сноски, бот не
будет делать из нее `<img>`, `<video>` или `<audio>`: она превратится в обычный
alt-текст. Это сделано специально, потому что Telegram Rich Message ожидает
медиа отдельными блоками.

Поддерживаемые расширения:

- картинки: `jpg`, `jpeg`, `png`, `webp`, `heic`, `heif`;
- видео/GIF: `gif`, `mp4`, `mov`, `m4v`, `webm`;
- аудио: `mp3`, `ogg`, `oga`, `m4a`, `aac`, `wav`, `flac`.

Бот определяет тип только по расширению в URL path. Ссылки без расширения и
опасные форматы вроде `svg`, `html`, `pdf`, `exe`, `js`, архивов и плейлистов не
вставляются в Rich Message.

## Когда Нужны Домен И Caddy

Бот работает через Telegram polling, поэтому webhook для Telegram ему не нужен.
Домен и Caddy нужны не для самого бота, а для картинок, которые лежат внутри
Markdown-файла байткодом.

Так обычно происходит с Google Docs: когда вы скачиваете документ как Markdown,
встроенные изображения могут попасть прямо внутрь `.md` файла как `data:` /
base64. Telegram Rich Message не принимает такие inline base64 картинки
напрямую. Поэтому Verstun достает картинку из файла, сохраняет ее на вашем
сервере и вставляет в Rich Message обычную HTTPS-ссылку:

```text
https://your-domain.com/media/<sha256>.<ext>
```

Когда Telegram создает Rich Message, серверы Telegram должны скачать картинку
по публичному HTTPS URL. Поэтому нужен домен или поддомен, DNS A-record на ваш
сервер и валидный TLS-сертификат.

Caddy в `docker-compose.yml` решает две задачи:

- автоматически выпускает и продлевает HTTPS-сертификат Let's Encrypt;
- отдает файлы из Docker volume `media` по маршруту `/media/*`.

Backend не публикуется наружу отдельным портом. Снаружи открыт только Caddy на
80/443, а внутри compose-сети он проксирует `/health` в backend и отдает
`/media/*` как статические файлы.

Если вы пишете обычный Markdown, где картинки уже указаны как публичные ссылки
вроде `https://example.com/image.png`, то Verstun не скачивает и не хостит эти
картинки. Он передает такие ссылки в Rich Message как есть. Для таких картинок
не нужен локальный `/media/*`: Telegram сам заберет их по уже готовым внешним
URL.

То же правило действует для внешних GIF, видео и аудио: бот не делает `GET`,
`HEAD`, не скачивает файл, не проверяет MIME и не становится CDN. Он только
проверяет схему `http/https`, расширение в URL path и вставляет безопасный
media-тег в Rich Message. Если по ссылке на самом деле лежит другой файл,
Telegram может не показать его или отклонить Rich Message.

Важно: штатная Docker-конфигурация в этом репозитории рассчитана на production
запуск и все равно требует `MEDIA_BASE_URL=https://...`, потому что она должна
безопасно поддерживать Google Docs exports со встроенными картинками. Если вы
точно не используете встроенные картинки и хотите запуск без домена/Caddy, это
уже advanced/dev сценарий: запускайте backend отдельно с `ENVIRONMENT=dev` и
публичными `https://...` ссылками на картинки в Markdown.

## Требования

- Linux-сервер или VPS с публичным IP.
- Домен или поддомен, направленный A-record на этот сервер.
- Открытые порты 80 и 443.
- Docker и Docker Compose.
- Telegram bot token из `@BotFather`.
- Ваш числовой Telegram user ID, например из `@userinfobot`.

## Установка

Перед запуском настройте DNS: A-record вашего домена или поддомена должен
указывать на публичный IP сервера. Дождитесь применения DNS и проверьте:

```bash
dig +short your-domain.com
curl -I http://your-domain.com
```

Если `dig` не показывает IP сервера или `curl` не доходит до Caddy, Caddy не
сможет выпустить HTTPS-сертификат. Проверьте DNS, firewall и доступность портов
80/443 извне.

Склонируйте репозиторий:

```bash
git clone https://github.com/pavel-molyanov/verstun.git
cd verstun
```

Создайте локальные конфиги:

```bash
cp .env.example .env
cp Caddyfile.example Caddyfile
chmod 600 .env
```

Отредактируйте `.env`:

```env
BOT_TOKEN=<botfather-token>
OWNER_TELEGRAM_ID=<telegram-user-id>
MEDIA_BASE_URL=https://your-domain.com/media
DATA_DIR=/srv/data
MEDIA_DIR=/srv/media
MEDIA_MAX_BYTES=1073741824
ENVIRONMENT=prod
```

Замените `your-domain.com` в `Caddyfile` на ваш домен:

```caddyfile
your-domain.com {
	encode zstd gzip

	handle /health {
		reverse_proxy backend:8000
	}

	handle_path /media/* {
		root * /srv/media
		file_server
		header X-Content-Type-Options nosniff
		header Content-Security-Policy "default-src 'none'; img-src 'self'; sandbox"
	}

	handle {
		respond "not found" 404
	}
}
```

Запустите:

```bash
docker compose up -d --build
```

Проверьте, что backend живой:

```bash
curl https://your-domain.com/health
```

Ожидаемый результат - JSON со статусом `ok` в нормальном состоянии. Если Caddy
еще выпускает сертификат, подождите минуту и повторите проверку.

Если не работает:

```bash
docker compose ps
docker compose logs -f caddy
docker compose logs -f backend
```

Типовые причины: домен в `Caddyfile` не совпадает с `MEDIA_BASE_URL`, DNS еще не
указывает на сервер, закрыты порты 80/443, неверный `BOT_TOKEN`, неверный
`OWNER_TELEGRAM_ID`, или backend не может писать в `DATA_DIR` / `MEDIA_DIR`.

## Первый Запуск В Telegram

Пишите боту с того Telegram-аккаунта, числовой ID которого указан в
`OWNER_TELEGRAM_ID`. Если бот молчит, сначала проверьте эту переменную и логи
backend: неизвестные пользователи игнорируются.

1. Откройте своего бота в Telegram и отправьте `/start`.
2. Отправьте `/demo`, чтобы проверить, что Rich Messages работают.
3. Создайте тестовый документ в Google Docs.
4. Скачайте его как `Markdown (.md)`.
5. Отправьте `.md` файл боту.

Бот должен вернуть Rich Message в личный чат. Если в документе были встроенные
картинки, проверьте, что они отображаются. Если картинок нет, домен и Caddy все
равно полезны для будущих постов, но простой текстовый Markdown может работать
без фактического обращения к `/media/*`.

## Команды

- `/start` - краткое приветствие.
- `/demo` - пример Rich Message со всеми основными элементами форматирования.
- `/channels` - подключение и отключение каналов.
- `/users` - управление пользователями и их доступами к каналам.

Команды управления работают в личном чате с ботом. Не используйте бота как
группового участника для подготовки постов.

## Пользователи

`OWNER_TELEGRAM_ID` - владелец инстанса. Владелец всегда имеет полный доступ:
может отправлять `.md`, подключать каналы, публиковать в любой подключенный
канал и управлять пользователями через `/users`.

Через `/users` владелец добавляет доверенных людей. Добавленный пользователь
может отправлять `.md` файлы боту. Публиковать через кнопки он сможет только в
те каналы, к которым владелец выдал доступ.

Неизвестные пользователи игнорируются. Это не публичный SaaS-бот, а
owner-managed self-host инструмент.

Ограничения доступа относятся только к публикации через самого бота. Если
человек получил готовый Rich Message в личном чате, Telegram может позволить ему
переслать или скопировать его туда, где у него уже есть собственные права.

## Каналы

Чтобы подключить канал:

1. Добавьте бота в канал администратором.
2. Дайте боту право публиковать сообщения.
3. Отправьте боту `/channels`.
4. Нажмите кнопку подключения нового канала.
5. Перешлите боту любой пост из нужного канала.

Бот проверит, что он действительно администратор канала и может публиковать
сообщения. После этого канал появится в списке.

Пересылать нужно именно пост из канала в личный чат с ботом. Если Telegram не
передает идентификатор канала из-за настроек приватности или ограничений
пересылки, бот не сможет подключить канал. В таком случае временно используйте
обычный тестовый пост из канала и проверьте настройки пересылки в Telegram.

Когда автор отправляет `.md`, бот возвращает Rich Message и отдельные кнопки
публикации. Кнопка канала показывается только если:

- канал подключен;
- текущему пользователю выдан доступ к этому каналу;
- у бота прямо сейчас есть право публикации в канале.

Кнопки остаются активными после публикации, поэтому один и тот же подготовленный
пост можно отправить в несколько каналов. Старые кнопки отключаются, когда в
этом личном чате создается новый пост из следующего `.md`.

Можно не подключать каналы вообще. Тогда используйте бота только как генератор
Rich Message: получите пост в личном чате и перешлите его вручную.

## Безопасность

Боту нужны права администратора в канале только для прямой публикации через
кнопки. Давайте минимум прав:

- разрешите публикацию сообщений;
- не давайте редактирование, удаление, добавление администраторов, приглашение
  пользователей и другие лишние права;
- добавляйте в `/users` только людей, которым доверяете;
- выдавайте каждому пользователю доступ только к нужным каналам.

Bot token - это секрет. Если у бота есть право публиковать в канале, человек с
bot token или доступом к серверу потенциально может публиковать в этот канал.
Никому не показывайте `.env`, не отправляйте токен в чаты, не коммитьте его в
git и не вставляйте в issue/лог/скриншот.

Ботов с правами на публикацию периодически взламывают через утечки токенов,
скомпрометированные серверы или лишние доступы. Практические меры:

- используйте отдельного бота только для Verstun;
- регулярно ротируйте token через `@BotFather`;
- сразу ротируйте token, если есть подозрение на утечку;
- ограничьте SSH-доступ к серверу;
- обновляйте Docker images и сервер;
- храните `.env` с правами `600`;
- для максимальной безопасности забирайте у бота право публикации, когда оно не
  нужно.

Самый строгий режим: дали боту право публиковать, отправили пост, забрали право
обратно. Тогда даже при последующей компрометации токена злоумышленник не сможет
публиковать в канал через этого бота.

Маршрут `/media/*` публичный. Все встроенные изображения из отправленных
Markdown-файлов сохраняются в Docker volume `media` и становятся доступны по
HTTPS-ссылке. Не отправляйте боту секретные изображения, персональные документы,
скриншоты с токенами или приватные черновики. Если ссылка на файл утекла, его
можно скачать без авторизации. Для очистки media-хранилища удалите ненужные
файлы из volume `media`; не используйте `docker compose down -v`, если не хотите
одновременно удалить состояние бота.

Если нашли уязвимость, сообщите приватно через GitHub Security Advisory или
личный контакт автора. Не публикуйте рабочие детали эксплуатации до исправления.

## Обновление

Перед обновлением сделайте резервную копию `data` volume: там лежат подключенные
каналы, пользователи и доступы. Не запускайте `docker compose down -v`, если не
хотите удалить состояние и media-файлы.

```bash
git pull
docker compose up -d --build
```

Если менялись настройки, проверьте `.env` и `Caddyfile`, затем снова откройте
`/health`.
